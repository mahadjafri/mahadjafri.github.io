Conversational Analytics Agent

A natural-language interface over the company's warehouse that lets non-technical stakeholders ask business questions in plain English — replacing most ad-hoc analyst tickets.

01The problem

Operations, marketing, and finance leaders had urgent questions about live metrics every day, but every question turned into a ticket queued behind a small analytics team. The bottleneck wasn't the data — it was the translation step between business question and SQL.

02The approach

I built an LLM-orchestrated agent that grounds itself in a curated semantic layer of the warehouse: vetted metrics, joins, and business definitions. Rather than letting the model write arbitrary SQL, the agent composes queries from a constrained set of validated primitives, runs them, interprets results, and follows up with clarifications.

Every answer surfaces the underlying query and assumptions, so users can verify before they act on a number.

03What I built

  • Semantic-layer-grounded agent that composes queries from validated metric primitives instead of free-form SQL
  • Retrieval pipeline over metric definitions, dashboards, and historical analyst answers
  • Evaluation harness with held-out business questions, scored on answer correctness and SQL validity
  • Slack and web interfaces for the same backing service
  • Feedback loop where flagged answers feed back into the semantic layer as new validated metrics

05Stack

LangChain OpenAI Python FastAPI PostgreSQL pgvector Slack API

06What I learned

Text-to-SQL works in demos and breaks in production. What actually shipped was text-to-structured-intent, where the model picks from a small library of approved metrics rather than inventing joins. The semantic layer was the product; the LLM was the front door.
